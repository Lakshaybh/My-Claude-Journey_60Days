Summary
The Hospital Management System is a web-based mini project (BCS554) developed by Lakshay Sharma, Mahek, and Achala Singh under the guidance of Mr Sanjay Kumar (Assistant Professor, Dept-CSE DS) at KCC Institutes of Technology and Management, Greater Noida, for the Academic Session 2025-26. The system is designed to manage hospital operations for small and medium healthcare facilities, including patient management, doctor management, appointment scheduling, billing, and data analytics. It uses HTML, CSS, JavaScript (frontend), Python + Flask (backend), Excel (hospital_data.xlsx) as a lightweight database, Pandas for data analysis, and Chart.js for visualizations. The project follows the Agile Iterative Development Model and was implemented in 4 phases: Planning & Design, Backend Development, Frontend Development, and Integration & Testing. The system is declared complete as of 12/01/2026.
Key Takeaways
🏗️
3-Layer Architecture
Presentation Layer (HTML5, CSS3, JS ES6, Chart.js) → Application Layer (Flask Port 5000, REST APIs) → Data Layer (hospital_data.xlsx with 5 sheets: Patients, Doctors, Appointments, Billing, Users).
🗄️
Excel as Database
Primary data stored in hospital_data.xlsx. Managed via excel_handler.py. 4 core entities: Patients, Doctors, Appointments, Billing. No enforced foreign-key constraints — code validates references manually.
🐍
Backend — Python Flask
app.py (entry point), analytics.py (metrics engine), excel_handler.py (read/write helpers), create_excel_db.py (seed/init utility). REST endpoints: /api/patients, /api/doctors, /api/appointments, /api/billing*, /api/login.
🌐
Frontend — 8 HTML Pages
index.html, login.html, dashboard.html, analytics.html, patients.html, doctors.html, appointments.html, billing.html. Styled with css/style.css, behavior via js/script.js (AJAX + form handling).
📊
Analytics Module
4 charts: Disease-wise Patient Distribution (bar), Gender-wise Patient Distribution (doughnut/pie), Monthly Patient Admission Trend (line), Department-wise Revenue Analysis (pie). Powered by Chart.js + Pandas.
🔐
Role-Based Login
3 user roles with sample credentials: Admin / Admin123 (Administrator), Doctor / Doctor123 (Doctor), Receptionist / Reception123 (Receptionist). Demographic credentials added for sample login activity.
⚙️
Development Model
Agile Iterative Development Model. 5 stages: Requirement Gathering → System Design → Implementation → Testing and Review → Deployment and Maintenance. Each module developed and tested separately.
🧪
Testing Methodology
Unit Testing (individual functions and APIs), Integration Testing (frontend-backend communication), System Testing (complete workflow), Acceptance Testing (usability and requirements verification).

Risks / Blockers
🔴 Excel concurrency — no concurrent write support
Excel files do not handle concurrent writes. If multiple users (Admin, Doctor, Receptionist) are logged in simultaneously and write data, corruption or data loss may occur. The report itself acknowledges: "Consider a locking strategy (simple file lock) if multiple processes may edit files concurrently."
🔴 No enforced referential integrity
There are no database-level foreign-key constraints in Excel. The code must validate references manually (e.g., when creating an Appointment, ensure patient_id and doctor_id exist). A missed validation could create orphaned records.
🟠 Hardcoded plaintext credentials in Excel (Users Sheet)
Passwords (Admin123, Doctor123, Reception123) are stored in the Excel Users Sheet. There is no mention of hashing or encryption, making it unsuitable for any real-world or production deployment.
🟠 Declaration page has unfilled placeholder text
The literal text "Guide name" and "...... Semester" remain in the declaration page as unfilled placeholders. This may affect academic submission acceptability.
🟠 No backup strategy implemented
The report recommends "periodically snapshot the Excel file(s)" and that "automated backups are recommended before bulk operations" — but no backup mechanism is implemented in the current system.
🟢 Scalability ceiling of Excel-based storage
Excel is acknowledged as a lightweight solution suitable for small-scale use. For any production use with hundreds of records or concurrent users, migration to SQLite or MySQL/PostgreSQL is required.

Additional Notes
Database Entity Relationships
Patient ↔ Appointments: One-to-many. One Patient may have many Appointments. Appointments carry patient_id.

Doctor ↔ Appointments: One-to-many. One Doctor can have many Appointments. Appointments carry doctor_id.

Appointment ↔ Billing: One-to-one (usually). A completed Appointment produces a Billing record referencing appointment_id and patient_id.

Referential integrity: No enforced database-level foreign-key constraint in Excel. Code must validate references. Recommended: add helper functions in excel_handler.py for common checks.
Research Gap Identified (Literature Review)
Gaps identified in existing Hospital Management Systems:

1. Lack of simple and lightweight systems for small hospitals
2. Limited integration of data analytics in traditional systems
3. Over-dependence on complex databases for small-scale applications
4. Poor focus on user-friendly and visually appealing interfaces

Traditional systems commonly use MySQL or Oracle and require complex setup and technical expertise, making them unsuitable for small organizations.
Future Scope Summary
1. Integration with MySQL or PostgreSQL for large data volumes, referential integrity, concurrent access
2. Advanced user authentication and role-based access control (beyond current 3-role demo)
3. Online appointment booking and patient portals (self-registration, history, medical info access)
4. Online payment gateway integration for digital billing
5. Cloud platform deployment for remote access, backup, and high availability
6. Machine learning for predicting patient inflow, disease trends, resource requirements
7. Mobile application for doctors and staff
8. Integration with government health systems and Electronic Health Records (EHR)
