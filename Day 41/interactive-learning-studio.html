MODULE 1

Foundations — Lists & Tuples

The building blocks of Python sequences: ordered, indexable collections — one mutable, one not.

📖 What Are Lists and Tuples?

A list is an ordered, mutable collection: fruits = ["apple", "banana", "cherry"]. You can add, remove, or change items after creation.

A tuple is an ordered, immutable collection: coords = (10, 20). Once created, its contents can never change.

🧩 Analogy: A list is like a whiteboard to-do list — you erase, add, and reorder items freely. A tuple is like a engraved plaque — once carved, the words are permanent. Use a plaque for facts that shouldn't change (like coordinates or a date of birth); use a whiteboard for things in flux (like a shopping cart).

💻 Indexing & Slicing

Both support zero-based indexing and slicing:

fruits = ["apple", "banana", "cherry", "date"]
fruits[0]      # "apple"
fruits[-1]     # "date"  (negative index = from the end)
fruits[1:3]    # ["banana", "cherry"]  (slice: start inclusive, end exclusive)
fruits[::-1]   # ["date", "cherry", "banana", "apple"]  (reversed)

🚫 Common Misconceptions

Myth: "Tuples can't contain mutable data."
Truth: A tuple can hold a list — the tuple itself is fixed, but a mutable element inside it can still change.
Myth: "Slicing a list creates a reference to the original."
Truth: Slicing creates a new list (a shallow copy); it doesn't share memory with the original list object.
Myth: "(1) is a tuple."
Truth: That's just an integer in parentheses. A single-item tuple needs a trailing comma: (1,).

🏋️ Practical Exercise

✅ Key Takeaways

  • Lists are ordered & mutable; tuples are ordered & immutable.
  • Negative indices count from the end; slicing returns a new object.
  • Tuples are hashable (when their contents are) so they can be dict keys or set members — lists cannot.
  • Choose tuples for fixed, "record-like" data; lists for collections that grow/shrink/change.
MODULE 2

Practical Application — Dictionaries & Sets

Key-value mappings and unique unordered collections, both powered by hashing under the hood.

📖 Dictionaries: Key → Value Mapping

A dict stores data as key-value pairs: user = {"name": "Ava", "age": 29}. Lookups by key are extremely fast — O(1) on average — because Python uses a hash table internally.

🧩 Analogy: A dictionary is like a real phone book — you look up a name (key) to instantly get a phone number (value), instead of scanning every entry one by one like you would with a list.
user = {"name": "Ava", "age": 29}
user["age"]            # 29
user["email"] = "ava@mail.com"   # add new key
user.get("phone", "N/A")         # safe lookup with default
for key, value in user.items():
    print(key, value)

📖 Sets: Unique, Unordered Collections

A set stores unique, unordered items: tags = {"python", "web", "python"} → {"python", "web"}. Great for membership tests and eliminating duplicates.

🧩 Analogy: A set is like a guest list at a private club — either your name is on it or it isn't, there are no duplicates, and the club doesn't care about the order names were added.

🚫 Common Misconceptions

Myth: "Dicts are unordered."
Truth: Since Python 3.7, dicts preserve insertion order as a language guarantee.
Myth: "You can use a list as a dict key."
Truth: Dict keys must be hashable; lists are mutable and unhashable, so only tuples, strings, and numbers work as keys.
Myth: "Sets can hold any items, like lists."
Truth: Set elements must be hashable too — no lists or dicts inside a set.

🏋️ Practical Exercise

✅ Key Takeaways

  • Dicts map unique keys to values with O(1) average lookup via hashing.
  • Sets guarantee uniqueness and support fast membership tests & math-style operations.
  • Only hashable (immutable) types can be dict keys or set members.
  • Dicts preserve insertion order since Python 3.7; sets do not guarantee order.
MODULE 3

Application — Comprehensions

Pythonic, compact syntax for building lists, dicts, sets, and generators from iterables.

📖 The Comprehension Pattern

A comprehension builds a new collection from an existing iterable in a single readable expression, replacing verbose loops:

# Traditional loop
squares = []
for n in range(10):
    squares.append(n ** 2)

# List comprehension — equivalent, more Pythonic
squares = [n ** 2 for n in range(10)]

# With a filter condition
evens_sq = [n ** 2 for n in range(10) if n % 2 == 0]

# Dict comprehension
square_map = {n: n ** 2 for n in range(5)}

# Set comprehension
unique_lengths = {len(word) for word in ["hi","bye","yo","cat"]}

# Generator expression (lazy, memory-efficient)
gen = (n ** 2 for n in range(1000000))
🧩 Analogy: A comprehension is like ordering a custom sandwich in one sentence — "give me turkey, no tomato, on wheat" — instead of describing every assembly step to the chef individually (loop, condition, append).

🚫 Common Misconceptions

Myth: "Comprehensions are always faster than loops."
Truth: They're often faster due to internal optimizations, but readability matters more — an overly nested comprehension can hurt both speed and clarity.
Myth: "(x for x in it) is a tuple comprehension."
Truth: Parentheses create a generator expression, not a tuple — there's no such thing as a tuple comprehension.
Myth: "Nesting comprehensions is always better than a loop."
Truth: Deeply nested comprehensions become unreadable; a plain loop is often clearer for complex logic.

🏋️ Practical Exercise

✅ Key Takeaways

  • Comprehensions replace loop+append boilerplate with a single expressive line.
  • List/dict/set comprehensions use []/{}; generator expressions use () and are lazy.
  • Add an if clause to filter; keep nesting shallow for readability.
  • Generators save memory for large or infinite sequences since values are produced on demand.
MODULE 4

Mastery — Choosing & Combining Structures

Bringing it all together: performance trade-offs, nested structures, and real-world decision-making.

📖 The Decision Framework

Real Python programs combine structures. A common pattern: a list of dicts for records, or a dict of lists for grouping.

# List of dicts — like rows in a table
users = [
    {"name": "Ava", "age": 29},
    {"name": "Ben", "age": 34},
]

# Dict of lists — grouping by category
by_dept = {
    "engineering": ["Ava", "Ben"],
    "sales": ["Cleo"],
}

# Nested comprehension: flatten a dict of lists
all_names = [name for names in by_dept.values() for name in names]
🧩 Analogy: Think of it like organizing a filing cabinet: tuples are locked folders (fixed records), lists are open trays you can rearrange, dicts are labeled drawers you search by name, and sets are a "no duplicates allowed" stamp collection.

🚫 Common Misconceptions

Myth: "One structure is always 'the best'."
Truth: The right choice always depends on access pattern — lookups vs iteration vs order vs uniqueness.
Myth: "Converting a list to a set to dedupe always preserves order."
Truth: Sets don't guarantee order; use dict.fromkeys(list) to dedupe while preserving order.
Myth: "Nested data structures need custom classes."
Truth: Plain nested lists/dicts/tuples handle most real-world structured data just fine — reach for classes only when behavior, not just data, is needed.

🏋️ Practical Exercise

✅ Key Takeaways

  • Combine structures — lists of dicts, dicts of lists — to model real-world data.
  • Sets/dicts give O(1) average membership tests; lists give O(n) — choose accordingly.
  • dict.fromkeys() dedupes while preserving order; set() does not guarantee order.
  • Mastery is about matching the structure's guarantees to your problem's real needs.

📝 Summary Notes

  • Lists: ordered, mutable sequences — the default general-purpose container.
  • Tuples: ordered, immutable — used for fixed records and as hashable dict keys.
  • Dicts: hash-based key→value maps with O(1) average lookup and insertion order.
  • Sets: hash-based unique collections — best for membership tests and set algebra.
  • Comprehensions: concise, Pythonic construction of lists/dicts/sets/generators.
  • Real applications combine structures: lists of dicts, dicts of lists, nested comprehensions.
